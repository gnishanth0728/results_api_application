# 🌐 Introduction

In the previous chapter, we established a TCP connection using the three-way handshake.

At this point, the browser and server have a reliable communication channel.

However, Tomcat still doesn't know:

* Which URL was requested
* Which HTTP method is being used
* Which Controller should execute
* Which headers were sent

Only after the browser sends an **HTTP request** does Spring Boot know what work to perform.

This chapter follows the **exact HTTP request** generated by your React application.

# 🌍 What is HTTP?

**HTTP (HyperText Transfer Protocol)** is an application-layer protocol used by web browsers and servers to exchange information.

HTTP defines:

* How requests are formatted
* How responses are formatted
* How errors are reported
* How resources are identified

HTTP does **not** provide:

* Reliable delivery (TCP does)
* Routing (IP does)
* Physical transmission (Ethernet/Wi-Fi does)

Think of HTTP as the language spoken over a TCP connection.

# 📨 Raw HTTP Request

When React executes:

```javascript
axios.get("/students/1051110244")
```

the browser generates an HTTP request similar to:

```http
GET /students/1051110244 HTTP/1.1
Host: localhost:8080
User-Agent: Mozilla/5.0
Accept: application/json
Accept-Encoding: gzip, deflate
Connection: keep-alive
```

This is plain text.

HTTP is **human-readable**, unlike TCP or Ethernet.

---

# 📖 Anatomy of an HTTP Request

Every HTTP request contains three parts.

```text
+--------------------------------+
| Request Line                   |
+--------------------------------+
| Headers                        |
+--------------------------------+
| Blank Line                     |
+--------------------------------+
| Optional Body                  |
+--------------------------------+
```

---

# 🛣️ Request Line

The first line tells the server **what the client wants**.

Example:

```http
GET /students/1051110244 HTTP/1.1
```

It contains three fields:

```text
HTTP Method

↓

Resource

↓

HTTP Version
```

---

## HTTP Method

```text
GET
```

Meaning:

> Retrieve data.

Other common methods:

| Method | Purpose          |
| ------ | ---------------- |
| GET    | Read data        |
| POST   | Create data      |
| PUT    | Replace data     |
| PATCH  | Partially update |
| DELETE | Remove data      |

Your Student Results API currently uses **GET** because it only retrieves results.

---

## Request URI

```text
/students/1051110244
```

Tomcat extracts:

```text
/students/{rollNumber}
```

Spring later binds:

```text
1051110244
```

to:

```java
@PathVariable Long rollNumber
```

---

## HTTP Version

```text
HTTP/1.1
```

This tells Tomcat which HTTP specification the browser is using.

---

# 🏷️ HTTP Headers

Headers contain metadata about the request.

Example:

```http
Host: localhost:8080
Accept: application/json
Connection: keep-alive
User-Agent: Mozilla/5.0
```

Headers are **not** the application data—they describe the request.

---

## 🏠 Host Header

```http
Host: localhost:8080
```

Required in HTTP/1.1.

Tomcat uses it to identify which virtual host should handle the request.

---

## 🌐 Accept Header

```http
Accept: application/json
```

Meaning:

> "I prefer the response in JSON format."

Spring Boot uses this during content negotiation.

---

## 👤 User-Agent

```http
User-Agent: Mozilla/5.0
```

Identifies the client.

Servers often use this for:

* Analytics
* Browser compatibility
* Logging

---

## 🔄 Connection Header

```http
Connection: keep-alive
```

Instead of closing the TCP connection after every request, the browser requests that it remain open for future requests.

This avoids repeating the TCP handshake for every HTTP request.

---

# 📦 HTTP Body

A GET request usually has **no body**.

```http
GET /students/1051110244 HTTP/1.1
Host: localhost:8080

```

For POST requests:

```http
POST /students

{
    "firstName":"Nishanth"
}
```

The JSON appears after the blank line.

---

# 📏 Content-Length

When a body exists:

```http
Content-Length: 154
```

Tomcat reads exactly **154 bytes** from the TCP stream.

Without this header, the server wouldn't know where the request body ends.

---

# 🍪 Cookies

Cookies allow browsers to store small pieces of state.

Example:

```http
Cookie: SESSION=abc123xyz
```

Spring Security often uses cookies to maintain authenticated sessions.

---

# 🔐 Authorization Header

Many REST APIs authenticate using bearer tokens.

Example:

```http
Authorization: Bearer eyJhbGciOiJIUzI1...
```

Spring Security validates this token before the request reaches your Controller.

---

# 🍃 How Tomcat Parses HTTP

Linux delivers raw bytes to Tomcat through the connected socket.

Conceptually:

```text
TCP Socket

↓

Raw Bytes

↓

HTTP Parser

↓

Request Object

↓

DispatcherServlet
```

Tomcat parses:

* Method
* URI
* Headers
* Body
* HTTP version

It then creates an internal request object.

---

# 🎯 DispatcherServlet Receives the Request

Tomcat passes the parsed request to Spring MVC.

```text
Tomcat

↓

DispatcherServlet

↓

HandlerMapping

↓

StudentController
```

DispatcherServlet is the **front controller** of Spring MVC.

It decides which Controller should handle the request.

---

# 💡 Key Takeaways

✅ HTTP is an application-layer protocol built on top of TCP.

✅ Every HTTP request begins with a request line.

✅ Headers provide metadata about the request.

✅ GET requests usually contain no message body.

✅ Tomcat parses raw HTTP text into Java request objects.

✅ DispatcherServlet is the entry point into Spring MVC.

✅ Only after HTTP parsing does your `StudentController` execute.
